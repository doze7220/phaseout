# Phase Out: Cluster String - Architecture & Design Rules

最終更新: 2026-06-06 (v0.7.2 時点)

このドキュメントは、パズルゲーム『Phase Out: Cluster String』におけるシステム設計、状態管理、イベントフック順序、描画規則などを定義した絶対的なルールブック（Single Source of Truth）です。今後の機能拡張やAIエディタによるコード改修時は、必ずこの仕様を遵守してください。

## 1. プロジェクト概要と駆動方式

* **ランタイム:** Vanilla HTML + JavaScript + Canvas 2D API + DOM(CSS)
* **物理エンジン:** Matter.js
* **ハイブリッド駆動方式:**
  本プロジェクトのゲームループは、`requestAnimationFrame`（ブラウザの描画タイミングに同期）と、Matter.jsの `Engine.update` （物理演算のステップ更新）を協調させて駆動します。Canvasでの物理オブジェクトやエフェクト（パーティクル、レーザー等）の描画と、DOM要素（スコア、ライフゲージ、UI表示）への状態反映を毎フレーム同期させることで、滑らかで一貫性のあるゲーム体験を実現しています。

各ファイルは論理的なカテゴリに分類され、厳格に責務が分かれています。他のファイルの責務を侵食しないようにしてください。

### 【データ定義層】
| ファイル名 | 責務（何をするか） | やらないこと（禁止事項） |
| :--- | :--- | :--- |
| **`config.js`** | 画面サイズ、色定義、LIFE設定、スコア表示設定などの静的定数と、ゲーム状態(`GameState`)の定義・初期化を行う。 | ロジックの実行やUIの操作、DOMの取得を一切行わない。 |
| **`audioConfig.js`** | BGM、SE、VOICE等のサウンドパス、初期マスター音量等の定数定義を行う。 | サウンドの再生処理やDOMアクセスを行わない。 |

### 【システム・ロジック層】
| ファイル名 | 責務（何をするか） | やらないこと（禁止事項） |
| :--- | :--- | :--- |
| **`main.js`** | エントリポイント。シーン遷移のバインディング、ボタンのイベントリスナ登録、コンフィグモーダルの制御を行う。 | 物理演算の直接制御や複雑なゲームロジックを記述しない。 |
| **`logic.js`** | 宝石の接続（BFS探索）、消去判定、スコア計算、LIFE増減、ゲームオーバー判定などの中核となるゲームルールを処理する。 | 描画処理（Canvasのパス描画等）を直接記述しない。 |
| **`physics.js`** | Matter.jsのエンジン・ワールドの初期化、壁や床の生成、宝石（Body）の生成・追加、およびメインのゲームループ（カスタムループ）の管理を行う。 | 描画処理やスコア計算などのゲームルールを扱わない。 |
| **`score.js`** | `BigInt`を利用したスコア計算や、単位（万・億・兆等）を付与したHTML文字列のフォーマット処理を行う。 | DOMへの直接出力（`innerHTML`への代入）を行わない（文字列を返すのみ）。 |

### 【レンダリング・UI演出層】
| ファイル名 | 責務（何をするか） | やらないこと（禁止事項） |
| :--- | :--- | :--- |
| **`renderer.js`** | Canvas 2D APIを用いた宝石のカスタム描画(`beforeRender`等のフック)、アセットマネージャーによる画像管理を行う。 | DOM要素の操作（スコアテキストやゲージ操作以外）や状態(`GameState`)の変更を行わない。 |
| **`title-animation.js`** | タイトル画面独自の宝石落下および破砕アニメーション（ParticleManagerの専用インスタンス利用）を管理する。 | ゲーム本編のロジックや物理エンジンには一切干渉しない。 |
| **`effects.js`** | （Facade）演出APIを外部に提供し、内部の各マネージャークラスへ委譲する。 | 自身では複雑なロジックを持たず、ゲーム進行状態の変更も行わない。 |
| **`Visualizer.js`** | 獲得経験値効率の視覚的フィードバック（WAVE/BLOCK）のCanvas描画、およびデバッグオーバーレイの更新を行う。 | 物理演算への干渉や、描画以外のゲームルールの変更を行わない。 |
| **`ParticleManager.js`** | パーティクル・火花配列のカプセル化、座標更新およびZ-Indexレイヤ4の描画を行う。 | 物理オブジェクトの座標や `GameState` のコア変数を直接書き換えない。 |
| **`LaserEffect.js`** | 伝播レーザー配列のカプセル化、アニメーション進行、レイヤ3の加算合成描画を行う。 | 物理状態や `GameState` の直接更新を行わない。 |
| **`ScreenEffects.js`** | CSS画面揺れ、連鎖ポップアップ、レベルアップ表示など、DOM・最前面の演出を管理する。 | Canvasの宝石描画に直接干渉しない。 |
| **`SoundManager.js`** | Web Audio APIを利用した音声のキャッシュ、再生、ローパスフィルター等のサウンド制御を行う。 | UI（DOM）の操作やゲームロジックへの干渉を行わない。 |

## 3. Game State（状態管理）一覧

ゲームは以下の状態機械（ステートマシン）として定義・管理されます。

* **TITLE** (タイトル画面)
  - ゲーム開始前の状態。Canvas上で背景アニメーションなどが動く。
* **PLAYING** (通常プレイ中)
  - パズル実行中のメイン状態。ユーザーのタップ入力や物理演算がアクティブに処理される。
* **STASIS** (一時停止中)
  - コンフィグメニュー展開時、またはライフ0の演出時などに適用。物理演算（`Engine.update`）の呼び出しをスキップすることで物理挙動のみを完全に停止し、描画ループは維持する。
* **RESULT** (リザルト画面)
  - ゲームオーバー後の最終スコア表示状態。操作は受け付けない。
* **PHASE_BREAK** (※将来実装予定)
  - 7色連鎖達成時などに発動するフィーバー状態。特別なエフェクトやボーナスルールが適用される。

### グローバル状態（GameState）の主要プロパティ

`config.js` で定義される `GameState` シングルトンオブジェクトが全ゲーム状態を保持します。

| プロパティ | 型 | 内容 |
|-----------|-----|------|
| `actualScore` | BigInt | 実際の内部スコア（連鎖による指数関数的増加に対応） |
| `displayScore` | BigInt | ドラムロール表示中の描画用スコア |
| `life` / `maxLife` | Number | 現在のライフ値と最大ライフ値 |
| `exp` / `nextLevelExp` | Number | 現在の経験値と次レベルまでの必要経験値 |
| `totalExp` | Number | 累計獲得経験値（UIアニメーション追従などの基準として利用） |
| `level` | Number | 現在のゲームレベル |
| `isGameOver` | Boolean | ゲームオーバー状態のフラグ |
| `GEMS` | Array | 画面上の全宝石（物理Body）の参照リスト |
| `playTimeMs` | Number | プレイ中の累計経過時間（ミリ秒、ステイシス中等は停止） |
| `maxChain` | Number | 1プレイ中の最大連鎖数 |
| `maxScorePerTap` | BigInt | 1タップから派生した連鎖で獲得した最大のスコア |
| `maxChainPerColor` | Object | 各色ごとの最大連鎖数を保持するオブジェクト |
| `colorDestroyCounts` | Object | 各色ごとの累計破壊数を保持するオブジェクト |

※ エフェクト用の一時配列（パーティクル、レーザー等）は各Managerクラス内部にカプセル化されています。

## 4. ★絶対変更禁止：Update・イベントフック順序

Matter.jsのイベントフック（`Events.on`）に基づくゲームロジックの実行順序です。この順序はバグ防止のため絶対に変更しないでください。

1. **`beforeUpdate`**
   - タップ入力（ポインタ位置）の処理、引力（アトラクター）の適用、物理Bodyに対する力の加算。
2. **物理エンジンのステップ実行**
   - Matter.jsの内部処理（座標更新、衝突計算）。
3. **`collisionStart`**
   - 宝石同士や壁との衝突判定、連鎖起点の検出および発火処理。
4. **`afterUpdate`**
   - 連鎖のBFS探索、パーティクル座標の更新・寿命処理、スコア計算、LIFE増減、ゲームオーバー判定の処理。

## 5. ★絶対変更禁止：Draw順序とUIバインディング規則

### Canvas描画順序 (Z-Index)
Canvasの描画処理（`renderer.js` および `effects.js`）は、以下の順序で背後から手前へオーバーレイ描画されます。

1. **背景** (星空などのベースレイヤー)
2. **宝石オブジェクト** (カスタムレンダラーによる画像描画)
3. **レーザー・伝播線** (加算合成 `lighter` を適用して発光表現)
4. **破片・火花パーティクル**
5. **波紋・フラッシュ演出** (最前面のエフェクト)

### DOM/CSSバインディング規則
JavaScript内のデータ状態をDOMへ反映する際は、以下のルールを遵守します。

* **スコア表示:** ドラムロール中の `displayScore` を `drawScoreToCanvas()` に渡し、事前生成した「Canvasスプライトキャッシュ」を用いて、スケール自動調整付きでCanvasレイヤー上に描画する。
* **オーラカラー:** トップ破壊色のカラーコードをCSS変数 `--aura-color` に動的に流し込み、ヘッダー等のシャドウ色に反映させる。
* **ゲージ類 (LIFE/EXP):** ゲージ残量をSVGの `stroke-dashoffset` 属性などに変更・バインディングし、リアルタイムに追従させる。
* **フローティングテキスト:** `scoreSpriteCache` を用いて動的に文字スプライトを結合したCanvasを生成し、一時的なDOM要素として画面上にアニメーション表示する。

## 6. リファクタリング共通ルール（注意事項）

コードの品質とパフォーマンスを維持するため、以下のルールを厳守してください。

* **副作用の禁止:** Matter.jsの物理座標（`Body.position`）を描画ループ(`afterRender`など)内で直接書き換えないこと。位置の更新は必ず物理ループ(`Update`系イベント)で行う。
* **DOMアクセス最小化:** DOMへのアクセス（要素の取得やスタイルの変更）は必要最小限に抑え、描画負荷（リフロー/リペイント）を極力下げる。
* **コンテキストの保護:** Canvasの加算合成（`globalCompositeOperation = 'lighter'`）を使用する際は、必ず `ctx.save()` と `ctx.restore()` で囲み、コンテキスト状態を保護して他の描画に影響を漏らさないこと。
* **Single Source of Truth:** 状態管理（スコア、レベル、ライフなど）は、必ず一元化された変数（`GameState`オブジェクト）から参照・更新すること。
* **モジュール依存関係の厳守:** `index.html` では `main.js` のみを `<script type="module">` として読み込み、それ以外のファイルはESモジュール（`import / export`）の依存関係グラフによって解決させること。グローバル汚染を防ぐため、安易に `window` に変数を生やさない。

## 8. 今後のリファクタリング・分離候補

将来的な技術的負債を防ぐため、現在肥大化の兆しがある部分を分離候補として管理します。

| 分離候補・タスク | 概要 | 優先度 |
|-----|------|--------|
| `renderer.js` の分割 | スコアやテキスト（単位・数字）の事前生成およびCanvasレイアウト処理を、`ScoreRenderer.js`（または `TextRenderer.js`）として分離する。純粋な宝石の描画とUIテキストの描画責務を分ける。 | 高 |
| `ScreenEffects.js` の整理 | 現在ファイル内に同居している `GaugeManager`（LIFE/EXPゲージのSVGアニメーション・タイマー管理）を、独立したクラスファイルとして切り出す。 | 中 |
| `logic.js` の責務分割 | BFS（幅優先探索）による連鎖判定などのアルゴリズム部分と、経験値・レベルアップ・ゲームオーバー判定などの「ゲーム進行管理部分」を分離する。 | 中 |
| イベントリスナーの分離 | `main.js` に直書きされている各種ボタンのClickイベントやResizeイベントなどを、UIハンドラ専用のスクリプトへ分離する。 | 低 |

※ `effects.js` の分割は完了済。
