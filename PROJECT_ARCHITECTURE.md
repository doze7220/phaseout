# PHASE OUT: Cluster Stirring - Architecture & Design Rules

最終更新: 2026-06-11 (v0.11.7 時点)

このドキュメントは、パズルゲーム『PHASE OUT: Cluster Stirring』におけるシステム設計、状態管理、イベントフック順序、描画規則などを定義した絶対的なルールブック（Single Source of Truth）です。今後の機能拡張やAIエディタによるコード改修時は、必ずこの仕様を遵守してください。

## 1. プロジェクト概要と駆動方式

* **ランタイム:** Vanilla HTML + JavaScript + Canvas 2D API (DOM依存を排除し、UIも完全Canvas化)
* **物理エンジン:** Matter.js
* **ハイブリッド駆動方式:**
  本プロジェクトのゲームループは、`requestAnimationFrame`（ブラウザの描画タイミングに同期）と、Matter.jsの `Engine.update` （固定タイムステップのAccumulatorによる物理演算のステップ更新）を協調させて駆動します。Canvasでの物理オブジェクトやエフェクト（パーティクル、レーザー等）、UI（スコア、ボタン、モーダル等）の描画を全て毎フレーム同期させることで、滑らかで一貫性のあるゲーム体験を実現しています。

## 3. Game State（状態管理）一覧

ゲームは以下の状態機械（ステートマシン）および SceneManager のスタックによって管理されます。

* **BOOT** (起動・初期化画面)
  - アセット読み込み中および初回起動待機画面。ユーザーのタップ（ブラウザのAutoplayポリシー解除）によりタイトルへクロスフェードする。
* **TITLE** (タイトル画面)
  - ゲーム開始前の状態。Canvas上で背景アニメーションなどが動く。
* **PUZZLE (PlayScene)** (通常プレイ中)
  - パズル実行中のメイン状態。ユーザーのタップ入力や物理演算がアクティブに処理される。
* **STASIS** (一時停止中)
  - コンフィグメニュー展開時、またはライフ0の演出時などに適用。物理演算の呼び出しをスキップすることで物理挙動のみを完全に停止し、描画ループは維持する。
* **RESULT (ResultScene)** (リザルト画面)
  - ゲームオーバー後の最終スコア表示状態。操作は受け付けない。パフォーマンス負荷を下げるため、物理演算およびゲーム盤面の再描画は完全に停止（フリーズ）される。
* **PHASE_BREAK** (※将来実装予定)
  - 7色連鎖達成時などに発動するフィーバー状態。特別なエフェクトやボーナスルールが適用される。

### グローバル状態（GameState）の主要プロパティ

`config.js` で定義される `GameState` シングルトンオブジェクトが全ゲーム状態を保持します。

| プロパティ | 型 | 内容 |
|-----------|-----|------|
| `actualScore` | BigInt | 実際の内部スコア（連鎖による指数関数的増加に対応） |
| `displayScore` | BigInt | ドラムロール表示中の描画用スコア |
| `life` / `maxLife` | Number | 現在のライフ値と最大ライフ値 |
| `exp` / `nextLevelExp` | Number | 現在の経験値と次レベルまでの必要経験値 |
| `totalExp` | Number | 累計獲得経験値（UIアニメーション追従などの基準として利用） |
| `level` | Number | 現在のゲームレベル |
| `isGameOver` | Boolean | ゲームオーバー状態のフラグ |
| `GEMS` | Array | 画面上の全宝石（物理Body）の参照リスト |
| `playTimeMs` | Number | プレイ中の累計経過時間（ミリ秒、ステイシス中等は停止） |
| `maxChain` | Number | 1プレイ中の最大連鎖数 |
| `maxScorePerTap` | BigInt | 1タップから派生した連鎖で獲得した最大のスコア |
| `maxChainPerColor` | Object | 各色ごとの最大連鎖数を保持するオブジェクト |
| `colorDestroyCounts` | Object | 各色ごとの累計破壊数を保持するオブジェクト |

### 定数による軽量化制御 (AppConfig)

パフォーマンスチューニングのため、`AppConfig.EFFECT_LEVEL`（FULL/LITE/NONE）による3段階の描画負荷制御が実装されています。各描画クラスはこれを参照し、以下の自律的な最適化を行います。
* **FULL**: リッチな描画。レーザーのマルチストローク発光（shadowBlur撤廃）、波紋スプライトアニメーション、大量のパーティクル生成等を行う。
* **LITE**: モバイル等の基本設定。パーティクル発生数を半減（火花はカット）、レーザーの加算合成オフ・単一線化、背景のBGMシンクロオフ等を行う。パーティクルはスプライトキャッシュを利用した `drawImage` により高効率に描画される。
* **NONE**: 超軽量設定。波紋エフェクト・スタンプフラッシュ・画面揺れ等の全カット。SVGゲージアニメーションの無効化、レーザーの極細アルファ線化など。

※ エフェクト用の一時配列（パーティクル、波紋、レーザー等）は各Managerクラス内部にカプセル化されています。DOMへの要素追加（div等）によるリフローを防ぐため、エフェクトは原則Canvasへの直接描画（スプライトキャッシュ活用）として実装されます。

## 4. ★絶対変更禁止：Update・イベントフック順序

Matter.jsのイベントフック（`Events.on`）に基づくゲームロジックの実行順序です。この順序はバグ防止のため絶対に変更しないでください。

1. **`beforeUpdate`**
   - タップ入力（ポインタ位置）の処理、引力（アトラクター）の適用、物理Bodyに対する力の加算。
2. **物理エンジンのステップ実行**
   - Matter.jsの内部処理（座標更新、衝突計算）。
3. **`collisionStart`**
   - 宝石同士や壁との衝突判定、連鎖起点の検出および発火処理。
4. **`afterUpdate`**
   - 連鎖のBFS探索、パーティクル座標の更新・寿命処理、スコア計算、LIFE増減、ゲームオーバー判定の処理。

## 5. ★絶対変更禁止：Draw順序とUIバインディング規則

### 5.1. レイヤー構成（全12層レンダーパイプライン）

描画処理は `MasterRenderer.js` の単一フックによって以下の順序で強制的に行われる。DOMの `z-index` による重ね合わせは原則として使用しない。

*   **第1層 (BACKGROUND)：背景**（タイトル画面のオーディオビジュアライザなど）
*   **第2層 (GEMS)：物理オブジェクト**（宝石本体：`renderer.js`）
*   **第3層 (LASER)：レーザー・伝播**（加算合成 lighter）
*   **第4層 (FOREGROUND_EFFECTS)：破片・火花等**
*   **第5層 (FRONT_EFFECTS)：前面エフェクト**（ヒットフラッシュ等）
*   **第6層 (IN_GAME_POST_EFFECT)：盤面専用ポストエフェクト**（ステイシス、ピンチ等のヴィネット）
*   **第7層 (UI_BASE)：基本UI**（LIFE/EXPゲージ、ヘッダー等のHUD）
*   **第8層 (POPUP_TEXT)：ポップアップテキスト**（ダメージや連鎖数、スコアポップアップ等）
*   **第9層 (MODAL_UI)：モーダルUI**（ConfigSceneなどの加算シーン。展開時は背面のタップ判定を無効化する）
*   **第10層 (GLOBAL_POST_EFFECT)：画面全体ポストエフェクト**（シーン切り替えの暗転やワイプなど）
*   **第11層 (SYSTEM_TOP)：最前面UI、システムフィードバック**（タップ波紋エフェクト等）
*   **第12層 (DEBUG_OVERLAY)：デバッグ・オーバーレイ**（FPSメーター、ビジュアライザ統計情報等）

### 5.2. UIバインディングとDOM更新/CSSバインディング規則
JavaScript内のデータ状態をDOMへ反映する際は、以下のルールを遵守します。

* **ヘッダーUI一括描画:** スコア、タイマー、タイムコスト（減少レート）、タップコスト、およびRATEの表示は、すべてDOM要素を廃止し、`ScoreRenderer.js` を用いて単一のヘッダーCanvas（`ui-header-canvas`）へ毎フレーム一括で描画する。PC版・スマホ版ともに「2段組み・数値右揃え」の統一レイアウトが適用される。
* **オーラカラー:** トップ破壊色のカラーコードをCSS変数 `--aura-color` に動的に流し込み、ヘッダー等のシャドウ色に反映させる。
* **ゲージ類 (LIFE/EXP):** ゲージ残量をSVGの `stroke-dashoffset` 属性などに変更・バインディングし、リアルタイムに追従させる。LIFEゲージは濃い緑（回復時は明緑）、EXPゲージはLIFEの2/3の太さ（青色）で内周に隙間なく密着配置し、直感的な視認性と階層構造を確保する。
* **フローティングテキスト:** `scoreSpriteCache` を用いて動的に文字スプライトを結合したCanvasを生成し、一時的なDOM要素として画面上にアニメーション表示する。

### 数値・スコアの表記フォーマットと描画仕様
本プロジェクトでは、用途や端末幅に応じてスコア等のフォーマットや描画手法を明確に定義しています。描画処理は `score.js` の `generateScoreData` でトークン配列化された後、各描画モジュール（Canvas / DOM）へ渡される**パイプライン方式**に統一されています。

| 表示箇所 | 描画方式 | PC版 最大スライス桁数 | スマホ版 最大スライス桁数 | フォーマット仕様・備考 |
| :--- | :--- | :--- | :--- | :--- |
| **累計スコア (ヘッダー)** | Canvas (`ScoreRenderer`) | 23桁 | 12桁 | レスポンシブにスライスされ、漢字単位（万・億等）が追記される。 |
| **SCORE RATE (ヘッダー)** | Canvas (`ScoreRenderer`) | 23桁 | 9桁 | 累計スコアよりスマホ版の制限が厳しい（レイアウトの兼ね合い）。 |
| **獲得スコア (ポップアップ)** | Canvas (`ScoreRenderer`) | 12桁 | 9桁 | タップ消去時のフローティング表示。縁取り付きで描画。 |
| **レベルアップ (RATE上昇演出)** | HTML (DOM) | 12桁 | 9桁 | `ScreenEffects.js` 経由。DOMアニメーション。漢字単位付き表示。 |
| **リザルト 累計スコア** | Canvas (`ScoreRenderer`) | 無制限 (自動改行) | 無制限 (自動改行) | 画面幅に応じて自動で折り返し、省略なしの全桁を描画する。 |
| **リザルト 1タップ最大スコア** | HTML (DOM) | 制限なし | 制限なし | リザルト画面のログ表示。省略なしの全桁表示。 |
| **連鎖数 (Chain)** | HTML (DOM) | - | - | 盤面の最大ブロック数が180程度のため、コンマ区切り等も行わずそのままの数値で表示。 |

## 6. リファクタリング共通ルール（注意事項）

コードの品質とパフォーマンスを維持するため、以下のルールを厳守してください。

* **副作用の禁止:** Matter.jsの物理座標（`Body.position`）を描画ループ(`afterRender`など)内で直接書き換えないこと。位置の更新は必ず物理ループ(`Update`系イベント)で行う。
* **DOMアクセス最小化:** DOMへのアクセス（要素の取得やスタイルの変更）は必要最小限に抑え、描画負荷（リフロー/リペイント）を極力下げる。
* **コンテキストの保護:** Canvasの加算合成（`globalCompositeOperation = 'lighter'`）を使用する際は、必ず `ctx.save()` と `ctx.restore()` で囲み、コンテキスト状態を保護して他の描画に影響を漏らさないこと。
* **Single Source of Truth:** 状態管理（スコア、レベル、ライフなど）は、必ず一元化された変数（`GameState`オブジェクト）から参照・更新すること。
* **モジュール依存関係の厳守:** `index.html` では `main.js` のみを `<script type="module">` として読み込み、それ以外のファイルはESモジュール（`import / export`）の依存関係グラフによって解決させること。グローバル汚染を防ぐため、安易に `window` に変数を生やさない。

## 9. マスターレンダーパイプライン (Canvas完全統合版)

すべての描画（物理オブジェクト、エフェクト、UI、モーダル、シーン遷移）は `MasterRenderer.js` の12層パイプラインで一元管理されます。DOMオーバーレイは廃止されました。

### 描画順序 (Layer ID)

1. **BACKGROUND (1):** 背景、グリッド、ビジュアライザ
2. **GEMS (2):** 宝石本体 (`renderer.js`)
3. **LASER (3):** レーザー・伝播エフェクト（加算合成）
4. **FOREGROUND_EFFECTS (4):** 波紋、爆発パーティクル、ヒットエフェクト
5. **FRONT_EFFECTS (5):** 前面エフェクト
6. **IN_GAME_POST_EFFECT (6):** 盤面専用ポストエフェクト（UIに被らない）
7. **UI_BASE (7):** ゲーム画面の基本UI（スコア、レベル、LIFEゲージ外枠など）
8. **POPUP_TEXT (8):** チェイン数、獲得スコアなどのポップアップテキスト
9. **MODAL_UI (9):** ConfigScene、リザルト画面、TITLEシーン、BOOTシーン
10. **GLOBAL_POST_EFFECT (10):** 画面全体フィルター・カラーオーバーレイ
11. **SYSTEM_TOP (11):** 画面最前面の波紋（タップフィードバック）
12. **DEBUG_OVERLAY (12):** FPSメーター、デバッグ情報など

FPSなどのシステムデバッグ表示は、レイヤー12にて最前面へ描画されます。

本プロジェクトでは、画面内に大量の剛体（Matter.jsのBody）が存在するため、物理演算の破綻（トンネリングやすり抜け、極端なめり込みによる暴発）を防ぐために以下の厳密なルールが設定されています。

### 9.1. FPS低下時のDeltaクランプ
* **問題点**: 画面描画の負荷などでFPSが低下すると、前フレームからの経過時間（Delta）が大きくなり、物理エンジンが「1フレームで物体が壁を突き抜けた」と誤認してしまいます。
* **対策**: `physics.js` のメインループ内で、Deltaの最大値を **33ms（30fps相当）** にクランプ（制限）しています。これにより、極端に処理落ちした場合でも物理エンジンは「ゆっくりとしたコマ送り」として処理され、すり抜けや理不尽な暴発を回避します。

### 9.2. ステージデータによる生成上限と形状制御（自己修復機能）
* **問題点**: 面積の大きいブロック（長方形など）が大量に生成されると、コンテナの容積を超過し、すり抜けや上に押し出される圧力が発生します。
* **対策**: `config.js` の `STAGE_DATA` において、ステージごとに「形状ごとの出現ウェイト」と「画面内上限数」を設定します。
  * デフォルト（STAGE_01）では、長方形の出現率を絞り、さらに三角（30）、四角（30）、長方形（10）と各ブロックに生成ハードリミットを設けています。
  * 上限に達した邪魔な形状はそれ以上降ってこなくなり、代わりに「上限無制限の丸（円）」に生成が置き換わります。
  * 円は面積が小さく角がないため、ゲームが長引いて盤面が詰まるほど自動的に流動性が高まり、物理エンジンへの負荷が下がるという「自己修復型の難易度・負荷調整システム」として機能します。

### 9.3. リザルト画面でのパフォーマンス保護（完全停止）
* **問題点**: ゲームオーバー時にリザルト画面へ移行した後も、裏で物理エンジンやCanvasの再描画が回り続けると無駄な負荷となり、FPSが極端に低下する。
* **対策**: `physics.js` 内のゲームループにて、`GameState.isGameOver` かつ `GameState.isStasis`（完全に時が止まった状態）の際は、物理更新（`Engine.update`）だけでなく盤面の再描画（`Render.world`）も完全にスキップして負荷をゼロにする制御を入れている。

## 10. 世界観設定および将来的なコアシステム構想（ロードマップ）

本作は単なるパズルゲームにとどまらず、エモーショナルな世界観とそれに紐づくカタルシスのあるシステム拡張が予定されています。今後の開発においては、以下のコンセプトを常に念頭に置いて実装を行います。

### 10.1. 世界観と「7つの色」の対立
* **完全な調和と熱的死（ステイシス）:** この世界において「完全な調和」とは、それ以上変化も拡張も起きない「死んだ世界（熱的死）」を意味する。パズルにおける一時停止状態が「STASIS」と呼ばれているのもこれに由来する。
* **プレイヤーの目的:** 世界が熱的死に至るのを防ぐため、プレイヤーは日々宝石（同色の塊）を破壊し、あえて「調和を崩す」ことで世界にエネルギーを与え、延命させている。
* **「分離された7色」:** 世界は7つの色に分類されている。各色は「自分が正義だ」と信じており、自分の色だけで世界を染めようと主張し合っている（パズル画面の「WAVE」モードは、各色が画面全体をカオスに専有しようと入り乱れるビジュアル表現となっている）。

### 10.2. ゲーム進行（レベルアップ）と難易度曲線
* **落下色数のレベル依存（未実装）:** ゲーム開始時は限られた色数からスタートする。プレイヤーが経験値を溜めてレベルアップするごとに、落下してくる色が1つずつ増えていく。
* **最終形態（カオス）:** レベルが進行し、最終的に7色すべてが画面内に入り乱れることで、パズルは最も混沌とした状態（最高難易度）となる。

### 10.3. カタルシス：「プリズムリンク」と「フェイズブレイク」
* **プリズムリンク（未実装）:** 7色全てが出揃う最高レベルに到達して初めて解放される究極の連鎖能力。本来は同色同士でしか繋がらない宝石が、**「赤 → 橙 → 黄 → 緑 → 水色 → 青 → 紫」**という虹の順序（スペクトル順）で色を跨いでリンク（チェイン）できるようになる。
* **フェイズブレイク / フィーバーモード（未実装）:** プリズムリンクを用いて7色すべてを一つに繋ぎ（一巡させ）ると、「フェイズブレイク」が発動する。対立していた7色が一つに統合されることで画面が「ホワイトアウト」し、圧倒的なスコアを獲得できるフィーバーモードへと突入する。
* **リバースリンクとブラックアウト（未実装）:** フェイズブレイク中のみ「プリズムリンク」は停止し、代わりにスペクトルの逆順で繋ぐことができる「リバースリンク」が発生する。これを達成すると「ブラックアウト」が発生し、「フェイズアウト」モードに突入する。これは熱的死に対抗する「宇宙の最終圧縮」、全てを飲み込むブラックホールを意図している。
* **フェイズアウト / 裏フィーバーモード（未実装）:** ブラックアウト後に到達する究極の裏モード。パズル盤面の物理法則自体が変化する。（※詳細は未定）
