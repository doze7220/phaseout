# PHASE OUT ∴ Cluster Stirring - Architecture & Design Rules
最終更新: 2026-06-21 (v0.26.21 時点)

このドキュメントは、パズルゲーム『PHASE OUT: Cluster Stirring』におけるシステム設計、状態管理、イベントフック順序、描画規則などを定義した絶対的なルールブック（Single Source of Truth）です。今後の機能拡張やAIエディタによるコード改修時は、必ずこの仕様を遵守してください。

## 0. 共通禁止項目
# ★Global Invariants（共通禁止・不変ルール）
本章はプロジェクト全体における最上位ルールであり、他のすべての仕様・設計・実装に優先する「不変条件（Invariants）」として扱う。
例外は許可するが、必ず実装前に事前申請、かつ仕様改訂として記録。

### 1. DOM操作の禁止領域
- ゲーム描画は完全にCanvas 2D APIへ統一する
- UI要素も含め、DOM描画は禁止する
- 一時的なデバッグ・開発用情報としてのDOM利用も厳禁とする
- 例外は「初期ロード・エントリポイント処理」のみとする
- 旧DOMベース実装（scene.js等）は拡張・再利用禁止

### 2. 描画責務の単一化
- 描画の最終制御は必ず `MasterRenderer.js` のみが行う
- レイヤー構造（12層）は固定であり変更禁止
- Scene / Logic / Manager から直接描画順序を操作することは禁止
- Canvas描画の呼び出し順序を外部から上書きしてはならない

### 3. 状態管理の単一化（Single Source of Truth）
- ゲーム状態は必ず `GameState` に集約する
- スコア・ライフ・レベル・進行状態の分散管理は禁止
- ローカル変数でゲーム進行を決定する設計は禁止
- 状態更新は必ず定義済みの管理層を経由すること

### 4. ロジックと描画の完全分離
- Renderer層はゲーム状態を変更してはならない
- Logic層はCanvas描画APIに直接アクセスしてはならない
- エフェクトは「表示専用」でありゲーム進行に干渉しない
- UI描画とゲーム進行ロジックの混在は禁止

### 5. 更新フロー（Update Pipeline）の固定
- Matter.jsイベント順序は以下を厳守すること：
  1. beforeUpdate
  2. Physics Step
  3. collisionStart
  4. afterUpdate
- この順序の変更は禁止
- `MasterRenderer.renderAll()` 内で状態更新処理を行ってはならない
- ゲームロジック更新は必ずメインループ経由で実行する

### 6. 物理演算の安全制約

- deltaTimeは最大33msにクランプすること（30fps相当上限）
- リザルト状態では物理更新を完全停止すること
- `Body.position` を描画処理側で直接変更してはならない
- 物理ワールドの破壊・再生成は専用管理層でのみ行う

### 7. 時間管理とアニメーションアーキテクチャ (Time & Animation Architecture)
- 時間は `realDelta` (実時間差分)、`gameDelta` (timeScale適用差分)、`gameTime` (累積ゲーム時間) の三系統で厳密に管理する。
- すべての `delta` は最上位ループから引数として下位へ流し渡すこと。末端コンポーネントが上位の管理層（MasterRenderer等）から `delta` を直接取得する逆依存は禁止する。
- 1回限りの演出（ポップアップ、レーザー、波紋等）は、必ず内部変数として `elapsed` を持ち、`update` にて `elapsed += gameDelta` する相対時間方式で管理すること。絶対時間方式への逆戻りは厳禁とする。
- `gameTime` の利用は、全体位相同期が必要な「周期運動（Math.sin 等）」のみに限定する。

### 8. Canvasコンテキスト保護ルール
- `ctx.save()` / `ctx.restore()` を必ずペアで使用すること
- globalCompositeOperationの状態を外部に漏らしてはならない
- Canvasのグローバル状態を暗黙に変更する実装は禁止
- 描画関数は副作用を持たないことが望ましい

### 9. モジュール責務侵食禁止
- 各モジュールは定義された責務範囲を逸脱してはならない
- Config / Logic / Scene / Render の境界を越えた実装は禁止
- `window` オブジェクトへのグローバル変数追加は禁止
- 他モジュール内部状態への直接アクセスは禁止

### 10. 例外禁止原則
- 本章のルールはすべて「絶対条件」とする
- パフォーマンス・仕様変更・バグ修正を理由とした例外は禁止
- 変更が必要な場合は「仕様改訂」として全体更新を行うこと

ゲーム用語およびゲームルールを定義する。
実装・改修時に必要となった場合のみ参照すること。

## 2. プロジェクト概要と駆動方式
* **ランタイム:** Vanilla HTML + JavaScript + Canvas 2D API (DOM依存を排除し、UIも完全Canvas化)
* **物理エンジン:** Matter.js
* **ハイブリッド駆動方式:**
  本プロジェクトのゲームループは、`requestAnimationFrame`（ブラウザの描画タイミングに同期）と、Matter.jsの `Engine.update` （固定タイムステップのAccumulatorによる物理演算のステップ更新）を協調させて駆動します。Canvasでの物理オブジェクトやエフェクト（パーティクル、レーザー等）、UI（スコア、ボタン、モーダル等）の描画を全て毎フレーム同期させることで、滑らかで一貫性のあるゲーム体験を実現しています。

## 4. Game State（状態管理）一覧

ゲームは以下の状態機械（ステートマシン）および SceneManager のスタックによって管理されます。

* **BOOT** (起動・初期化画面)
  - アセット読み込み中および初回起動待機画面。ユーザーのタップ（ブラウザのAutoplayポリシー解除）によりタイトルへクロスフェードする。
* **TITLE** (タイトル画面)
  - ゲーム開始前の状態。Canvas上で背景アニメーションなどが動く。
* **PUZZLE (PlayScene)** (通常プレイ中)
  - パズル実行中のメイン状態。ユーザーのタップ入力や物理演算がアクティブに処理される。
* **STASIS** (一時停止中)
  - コンフィグメニュー展開時、またはライフ0の演出時などに適用。物理演算の呼び出しをスキップする（`isPuzzlePaused`）ことで物理挙動のみを完全に停止し、フェイズ等のシステム進行（`isSystemPaused`）も停止する。
* **RESULT (ResultScene)** (リザルト画面)
  - ゲームオーバー後の最終スコア表示状態。操作は受け付けない。パフォーマンス負荷を下げるため、物理演算およびゲーム盤面の再描画は完全に停止（フリーズ）される。

### フェイズ管理（PhaseManager）

PlayScene内部のパズル進行状態は `PhaseManager.js` がステートマシンとして一元管理します。将来実装予定のフィーバーモード（白・黒等）の状態管理についても、システム全体のグローバル状態（GameStateやScene）としてではなく、PlayScene内部の `PhaseManager` の責務として完全に集約されます。
* **PHASE_START** (ゲーム開始待機中): パズル開始直後1.5秒間。入力および時間経過によるLIFE減少をブロックする。
* **PHASE_NORMAL** (通常パズル時): 通常のゲームプレイ状態。タップと連鎖、LIFE減少が発生する。
* **PHASE_GAMEOVER** (ゲームオーバー演出中): LIFE0到達時に移行。timeScaleを0.2に落としステイシスエフェクトを発動する。判定の詳細条件・キャンセル条件については `PROJECT_GLOSSARY_AND_GAMEDESIGN.md` を参照すること。
* **PHASE_WHITE_ENTER / PHASE_WHITE / PHASE_WHITE_EXIT** (ホワイトフェイズ関連): 将来実装予定のフィーバー状態等のステート定義。

#### ゲームオーバーキャンセルの実装ルール（v0.26.19以降）
ゲームオーバー判定の詳細仕様・条件は **`PROJECT_GLOSSARY_AND_GAMEDESIGN.md` を絶対資料** とする。両資料に矛盾が生じた場合は必ずGLOSSARYを優先すること。
実装上のルールとして以下を厳守する：
* ゲームオーバー状態のキャンセルは **必ず `PhaseManager.cancelGameOver()` を経由** して行うこと。
* `PhaseManager.js` の内部状態（`currentPhase`, `stateTimer`, `isFinalGameOverTriggered` 等）を外部から直接書き換えることは **Global Invariants 違反であり絶対禁止**。
* `cancelGameOver()` は `setGameOver()` が変更した全状態を一括で生存状態へ戻す責務を持つ。新たな状態変更が `setGameOver()` に加わった場合は、`cancelGameOver()` にも対応する復元処理を必ず追加すること。


###### 【将来拡張枠】フェイズ状態遷移の完全フロー
※ホワイトフェイズ～ブラックフェイズの完全実装・仕様確定後に、フェイズ間の移行条件と一方通行のフロー（例：NORMAL → WHITE_ENTER → WHITE...）をここに明記する予定。現時点では未定義とし、流動的な実装仕様に依存する。

###### 【将来拡張枠】特殊フェイズの絶対ルール（ホワイトフェイズ・ブラックフェイズ等）

### グローバル状態（GameState）の主要プロパティ

`config.js` で定義される `GameState` シングルトンオブジェクトが全ゲーム状態を保持します。

| プロパティ | 型 | 内容 |
|-----------|-----|------|
| `actualScore` | BigInt | 実際の内部スコア（連鎖による指数関数的増加に対応） |

| `life` / `maxLife` | Number | 現在のライフ値と最大ライフ値 |

### 定数による軽量化制御 (AppConfig)

パフォーマンスチューニングのため、`AppConfig.EFFECT_LEVEL`（FULL/LITE/NONE）による3段階の描画負荷制御が実装されています。各描画クラスはこれを参照し、以下の自律的な最適化を行います。
* **FULL**: リッチな描画。レーザーのマルチストローク発光（shadowBlur撤廃）、波紋スプライトアニメーション、大量のパーティクル生成等を行う。
* **LITE**: モバイル等の基本設定。パーティクル発生数を半減（火花はカット）、レーザーの加算合成オフ・単一線化、背景のBGMシンクロオフ等を行う。パーティクルはスプライトキャッシュを利用した `drawImage` により高効率に描画される。
* **NONE**: 超軽量設定。波紋エフェクト・スタンプフラッシュ・画面揺れ等の全カット。SVGゲージアニメーションの無効化、レーザーの極細アルファ線化など。

※ エフェクト用の一時配列（パーティクル、波紋、レーザー等）は各Managerクラス内部にカプセル化されています。DOMリフローを防ぐため、エフェクトは原則Canvasへの直接描画として実装されます。 

## 5. ★絶対変更禁止：Update・イベントフック順序

Matter.jsのイベントフック（`Events.on`）に基づくゲームロジックの実行順序です。この順序はバグ防止のため絶対に変更しないでください。

1. **`beforeUpdate`**
   - タップ入力（ポインタ位置）の処理、引力（アトラクター）の適用、物理Bodyに対する力の加算。
2. **Physics Step（Matter.js）**
   - Matter.jsの内部処理（座標更新、衝突計算）、ワールド状態更新。
3. **`collisionStart`**
   - 宝石同士や壁との衝突判定、連鎖起点の検出および発火処理。
4. **`afterUpdate`**
   - ChainAlgorithmによる連鎖のBFS探索、パーティクル座標の更新・寿命処理、スコア計算、LIFE増減、ゲームオーバー判定の処理。

**【禁止事項】** `MasterRenderer.js` の `renderAll()` 内部では純粋な描画（Contextの操作）のみを行い、状態更新のコールバック（`globalUpdateCallbacks` や `SceneManager.update` 等）を絶対に呼び出さないこと。更新処理は必ずメインループ(`loop`)内で `deltaTime` を渡して実行しなければならない（`NaN`汚染によるゲーム破壊を防ぐため）。

## 7. リファクタリング共通ルール（注意事項）

コードの品質とパフォーマンスを維持するため、以下のルールを厳守してください。

* **副作用の禁止:** Matter.jsの物理座標（`Body.position`）を描画ループ(`afterRender`など)内で直接書き換えないこと。位置の更新は必ず物理ループ(`Update`系イベント)で行う。
* **コンテキストの保護:** Canvasの加算合成（`globalCompositeOperation = 'lighter'`）を使用する際は、必ず `ctx.save()` と `ctx.restore()` で囲み、コンテキスト状態を保護して他の描画に影響を漏らさないこと。
* **Single Source of Truth:** 状態管理（スコア、レベル、ライフなど）は、必ず一元化された変数（`GameState`オブジェクト）から参照・更新すること。
* **モジュール依存関係の厳守:** `index.html` では `main.js` のみを `<script type="module">` として読み込み、それ以外のファイルはESモジュール（`import / export`）の依存関係グラフによって解決させること。グローバル汚染を防ぐため、安易に `window` に変数を生やさない。

## 9. 物理演算とパフォーマンス保護の絶対ルール
本作の大量の剛体（Matter.jsのBody）による物理破綻（トンネリングやすり抜け）を防ぐため、以下の安全装置はコード改修時も絶対に維持すること。

*   **Deltaクランプ (`physics.js`)**: FPS低下時のすり抜け暴発を防ぐため、物理更新の `deltaTime` は最大 **33ms**（30fps相当）にクランプする。
*   **自己修復型の形状制御 (`config.js`)**: 盤面圧迫を防ぐため、面積の大きいブロック（長方形等）には生成上限を設け、上限到達時は負荷の低い「円」へフォールバック生成させる。
*   **リザルト時の完全停止 (`physics.js`)**: パフォーマンス保護のため、`isGameOver` かつ `isPuzzlePaused` の間は、物理更新と盤面の再描画を完全にスキップする。
