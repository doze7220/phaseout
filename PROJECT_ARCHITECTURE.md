# Phase Out: Cluster Stirring - Architecture & Design Rules

最終更新: 2026-06-06 (v0.8.6 時点)

このドキュメントは、パズルゲーム『Phase Out: Cluster Stirring』におけるシステム設計、状態管理、イベントフック順序、描画規則などを定義した絶対的なルールブック（Single Source of Truth）です。今後の機能拡張やAIエディタによるコード改修時は、必ずこの仕様を遵守してください。

## 1. プロジェクト概要と駆動方式

* **ランタイム:** Vanilla HTML + JavaScript + Canvas 2D API + DOM(CSS)
* **物理エンジン:** Matter.js
* **ハイブリッド駆動方式:**
  本プロジェクトのゲームループは、`requestAnimationFrame`（ブラウザの描画タイミングに同期）と、Matter.jsの `Engine.update` （物理演算のステップ更新）を協調させて駆動します。Canvasでの物理オブジェクトやエフェクト（パーティクル、レーザー等）の描画と、DOM要素（スコア、ライフゲージ、UI表示）への状態反映を毎フレーム同期させることで、滑らかで一貫性のあるゲーム体験を実現しています。

## 3. Game State（状態管理）一覧

ゲームは以下の状態機械（ステートマシン）として定義・管理されます。

* **BOOT** (起動・初期化画面)
  - アセット読み込み中および初回起動待機画面。ユーザーのタップ（ブラウザのAutoplayポリシー解除）によりタイトルへクロスフェードする。
* **TITLE** (タイトル画面)
  - ゲーム開始前の状態。Canvas上で背景アニメーションなどが動く。
* **PLAYING** (通常プレイ中)
  - パズル実行中のメイン状態。ユーザーのタップ入力や物理演算がアクティブに処理される。
* **STASIS** (一時停止中)
  - コンフィグメニュー展開時、またはライフ0の演出時などに適用。物理演算（`Engine.update`）の呼び出しをスキップすることで物理挙動のみを完全に停止し、描画ループは維持する。
* **RESULT** (リザルト画面)
  - ゲームオーバー後の最終スコア表示状態。操作は受け付けない。
* **PHASE_BREAK** (※将来実装予定)
  - 7色連鎖達成時などに発動するフィーバー状態。特別なエフェクトやボーナスルールが適用される。

### グローバル状態（GameState）の主要プロパティ

`config.js` で定義される `GameState` シングルトンオブジェクトが全ゲーム状態を保持します。

| プロパティ | 型 | 内容 |
|-----------|-----|------|
| `actualScore` | BigInt | 実際の内部スコア（連鎖による指数関数的増加に対応） |
| `displayScore` | BigInt | ドラムロール表示中の描画用スコア |
| `life` / `maxLife` | Number | 現在のライフ値と最大ライフ値 |
| `exp` / `nextLevelExp` | Number | 現在の経験値と次レベルまでの必要経験値 |
| `totalExp` | Number | 累計獲得経験値（UIアニメーション追従などの基準として利用） |
| `level` | Number | 現在のゲームレベル |
| `isGameOver` | Boolean | ゲームオーバー状態のフラグ |
| `GEMS` | Array | 画面上の全宝石（物理Body）の参照リスト |
| `playTimeMs` | Number | プレイ中の累計経過時間（ミリ秒、ステイシス中等は停止） |
| `maxChain` | Number | 1プレイ中の最大連鎖数 |
| `maxScorePerTap` | BigInt | 1タップから派生した連鎖で獲得した最大のスコア |
| `maxChainPerColor` | Object | 各色ごとの最大連鎖数を保持するオブジェクト |
| `colorDestroyCounts` | Object | 各色ごとの累計破壊数を保持するオブジェクト |

### 定数による軽量化制御 (AppConfig)

パフォーマンスチューニングのため、`AppConfig.EFFECT_LEVEL`（FULL/LITE/NONE）による3段階の描画負荷制御が実装されています。各描画クラスはこれを参照し、以下の自律的な最適化を行います。
* **FULL**: リッチな描画。レーザーのマルチストローク発光（shadowBlur撤廃）、波紋スプライトアニメーション、大量のパーティクル生成等を行う。
* **LITE**: モバイル等の基本設定。パーティクル発生数を半減（火花はカット）、レーザーの加算合成オフ・単一線化、背景のBGMシンクロオフ等を行う。パーティクルはスプライトキャッシュを利用した `drawImage` により高効率に描画される。
* **NONE**: 超軽量設定。波紋エフェクト・スタンプフラッシュ・画面揺れ等の全カット。SVGゲージアニメーションの無効化、レーザーの極細アルファ線化など。

※ エフェクト用の一時配列（パーティクル、波紋、レーザー等）は各Managerクラス内部にカプセル化されています。DOMへの要素追加（div等）によるリフローを防ぐため、エフェクトは原則Canvasへの直接描画（スプライトキャッシュ活用）として実装されます。

## 4. ★絶対変更禁止：Update・イベントフック順序

Matter.jsのイベントフック（`Events.on`）に基づくゲームロジックの実行順序です。この順序はバグ防止のため絶対に変更しないでください。

1. **`beforeUpdate`**
   - タップ入力（ポインタ位置）の処理、引力（アトラクター）の適用、物理Bodyに対する力の加算。
2. **物理エンジンのステップ実行**
   - Matter.jsの内部処理（座標更新、衝突計算）。
3. **`collisionStart`**
   - 宝石同士や壁との衝突判定、連鎖起点の検出および発火処理。
4. **`afterUpdate`**
   - 連鎖のBFS探索、パーティクル座標の更新・寿命処理、スコア計算、LIFE増減、ゲームオーバー判定の処理。

## 5. ★絶対変更禁止：Draw順序とUIバインディング規則

### Canvas描画順序 (Z-Index)
Canvasの描画処理（`renderer.js` および `effects.js`）は、以下の順序で背後から手前へオーバーレイ描画されます。

1. **背景** (星空などのベースレイヤー)
2. **宝石オブジェクト** (スプライトキャッシュによるスタンプ描画)
3. **レーザー・伝播線** (FULL時:加算合成 `lighter` によるマルチストローク描画)
4. **破片・火花パーティクル** (スプライトキャッシュによるスタンプ描画)
5. **波紋・フラッシュ演出** (最前面Canvasエフェクト)

### DOM/CSSバインディング規則
JavaScript内のデータ状態をDOMへ反映する際は、以下のルールを遵守します。

* **スコア表示:** ドラムロール中の `displayScore` を `drawScoreToCanvas()` に渡し、事前生成した「Canvasスプライトキャッシュ」を用いて、スケール自動調整付きでCanvasレイヤー上に描画する。
* **オーラカラー:** トップ破壊色のカラーコードをCSS変数 `--aura-color` に動的に流し込み、ヘッダー等のシャドウ色に反映させる。
* **ゲージ類 (LIFE/EXP):** ゲージ残量をSVGの `stroke-dashoffset` 属性などに変更・バインディングし、リアルタイムに追従させる。
* **フローティングテキスト:** `scoreSpriteCache` を用いて動的に文字スプライトを結合したCanvasを生成し、一時的なDOM要素として画面上にアニメーション表示する。

## 6. リファクタリング共通ルール（注意事項）

コードの品質とパフォーマンスを維持するため、以下のルールを厳守してください。

* **副作用の禁止:** Matter.jsの物理座標（`Body.position`）を描画ループ(`afterRender`など)内で直接書き換えないこと。位置の更新は必ず物理ループ(`Update`系イベント)で行う。
* **DOMアクセス最小化:** DOMへのアクセス（要素の取得やスタイルの変更）は必要最小限に抑え、描画負荷（リフロー/リペイント）を極力下げる。
* **コンテキストの保護:** Canvasの加算合成（`globalCompositeOperation = 'lighter'`）を使用する際は、必ず `ctx.save()` と `ctx.restore()` で囲み、コンテキスト状態を保護して他の描画に影響を漏らさないこと。
* **Single Source of Truth:** 状態管理（スコア、レベル、ライフなど）は、必ず一元化された変数（`GameState`オブジェクト）から参照・更新すること。
* **モジュール依存関係の厳守:** `index.html` では `main.js` のみを `<script type="module">` として読み込み、それ以外のファイルはESモジュール（`import / export`）の依存関係グラフによって解決させること。グローバル汚染を防ぐため、安易に `window` に変数を生やさない。

## 9. 世界観設定および将来的なコアシステム構想（ロードマップ）

本作は単なるパズルゲームにとどまらず、エモーショナルな世界観とそれに紐づくカタルシスのあるシステム拡張が予定されています。今後の開発においては、以下のコンセプトを常に念頭に置いて実装を行います。

### 9.1. 世界観と「7つの色」の対立
* **完全な調和と熱的死（ステイシス）:** この世界において「完全な調和」とは、それ以上変化も拡張も起きない「死んだ世界（熱的死）」を意味する。パズルにおける一時停止状態が「STASIS」と呼ばれているのもこれに由来する。
* **プレイヤーの目的:** 世界が熱的死に至るのを防ぐため、プレイヤーは日々宝石（同色の塊）を破壊し、あえて「調和を崩す」ことで世界にエネルギーを与え、延命させている。
* **「分離された7色」:** 世界は7つの色に分類されている。各色は「自分が正義だ」と信じており、自分の色だけで世界を染めようと主張し合っている（パズル画面の「WAVE」モードは、各色が画面全体をカオスに専有しようと入り乱れるビジュアル表現となっている）。

### 9.2. ゲーム進行（レベルアップ）と難易度曲線
* **落下色数のレベル依存（未実装）:** ゲーム開始時は限られた色数からスタートする。プレイヤーが経験値を溜めてレベルアップするごとに、落下してくる色が1つずつ増えていく。
* **最終形態（カオス）:** レベルが進行し、最終的に7色すべてが画面内に入り乱れることで、パズルは最も混沌とした状態（最高難易度）となる。

### 9.3. カタルシス：「プリズムリンク」と「フェイズブレイク」
* **プリズムリンク（未実装）:** 7色全てが出揃う最高レベルに到達して初めて解放される究極の連鎖能力。本来は同色同士でしか繋がらない宝石が、**「赤 → 橙 → 黄 → 緑 → 水色 → 青 → 紫」**という虹の順序（スペクトル順）で色を跨いでリンク（チェイン）できるようになる。
* **フェイズブレイク / フィーバーモード（未実装）:** プリズムリンクを用いて7色すべてを一つに繋ぎ（一巡させ）ると、「フェイズブレイク」が発動する。対立していた7色が一つに統合されることで画面が「ホワイトアウト」し、圧倒的なスコアを獲得できるフィーバーモードへと突入する。
* **リバースリンクとブラックアウト（未実装）:** フェイズブレイク中のみ「プリズムリンク」は停止し、代わりにスペクトルの逆順で繋ぐことができる「リバースリンク」が発生する。これを達成すると「ブラックアウト」が発生し、「フェイズアウト」モードに突入する。これは熱的死に対抗する「宇宙の最終圧縮」、全てを飲み込むブラックホールを意図している。
* **フェイズアウト / 裏フィーバーモード（未実装）:** ブラックアウト後に到達する究極の裏モード。パズル盤面の物理法則自体が変化する。（※詳細は未定）
